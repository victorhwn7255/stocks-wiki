# KV Cache in Large Language Models

## Executive summary

[established] The key-value cache, usually shortened to **KV cache**, is the working memory that makes autoregressive transformer decoding practical. At each transformer layer, the model computes a **key** and **value** vector for every processed token; during generation, those vectors are stored so the next token only has to compute attention against the already-saved history instead of recomputing the entire prompt from scratch. Without KV caching, decoding latency would grow catastrophically because every new token would trigger a full re-run over all prior tokens across all layers. This mechanism is foundational to modern LLM serving, from open-source engines such as vLLM, SGLang, TensorRT-LLM, and Hugging Face TGI to commercial API features marketed as prompt caching or context caching by OpenAI, Anthropic, Google, and Fireworks. citeturn16search3turn13search0turn17search9turn10search2turn10search3turn31view2turn17search1

[established] The central operational split in LLM inference is **prefill** versus **decode**. Prefill ingests the prompt and builds the initial KV cache; because it processes many tokens in parallel with large GEMMs, it is typically **compute-bound**. Decode generates one or a few new tokens at a time; because each step repeatedly reads model weights and the ever-growing KV cache from memory, it is typically **memory-bandwidth-bound**. This is why the same GPU can look “fast” on short prompts yet bog down on long-context chat, agent loops, or multimodal workloads. Sources broadly agree on the compute-vs-bandwidth distinction, but they disagree on which phase is “the expensive part” because that depends on workload shape: long prompts with short answers make prefill dominate time-to-first-token, while very long generations or extreme context lengths can make decode dominate steady-state cost and latency. citeturn37search3turn37search22turn37search9turn37search0

[established] The industry’s immediate problem is not that KV cache exists, but that it scales linearly with **layers × KV heads × head dimension × sequence length × batch/concurrency × precision bytes**. Using Meta’s Llama 3.1 70B configuration as a concrete example—80 layers, 8 KV heads, head dimension 128—raw BF16 KV cache is about **320 KiB per token**. That is roughly **2.5 GiB at 8K**, **10 GiB at 32K**, and **40 GiB at 128K** for a single sequence, before allocator overhead and other runtime buffers. Ten simultaneous 128K sessions are therefore on the order of **400 GiB** of raw KV state. This is why H100/H200/MI300X class accelerators are increasingly sold on **memory capacity and bandwidth**, not just FLOPS. NVIDIA’s H200 ships with 141 GB HBM3e at 4.8 TB/s, and AMD’s MI300X with 192 GB HBM3 at 5.325 TB/s; both specs are best understood as responses to long-context inference pressure as much as to training demand. citeturn4view2turn19search0turn19search1

[evidence-backed inference] The state of the art is converging on a stack of complementary tactics rather than one silver bullet. At the **model level**, MQA and GQA reduce KV heads, while DeepSeek’s MLA compresses KV into a latent representation and officially reports a **93.3%** KV reduction versus its earlier 67B model. At the **systems level**, PagedAttention, RadixAttention, prefix caching, cache-aware routing, prefill/decode disaggregation, and KV offload to CPU/SSD/remote tiers turn KV from a local scratch buffer into a cluster resource. At the **compression level**, FP8 KV cache is already production-grade in major engines, while lower-bit schemes such as KIVI, KVQuant, and Google’s TurboQuant remain more frontier. The likely near-term equilibrium is “**architectural reduction first, software reuse second, quantization third, offload last**,” because that order usually gives the best quality–latency tradeoff. citeturn5view0turn13search0turn13search2turn29search0turn29search3turn26search8turn11search1turn11search0turn27search4

[evidence-backed inference] For industry structure, the durable bottlenecks are shifting. **HBM and accelerator memory bandwidth** remain the hottest bottleneck. **Serving software** is partly commoditizing at the single-node layer because open-source engines are strong, but scale-out KV reuse, offload, routing, observability, and multimodal support are becoming the higher-value control plane. **NAND/SSD** is the swing factor: vendor roadmaps from Samsung, Kioxia, Western Digital, and NVIDIA-linked storage initiatives are increasingly explicit about KV-cache offload and “context memory” as an AI-serving workload, but the degree of upside depends on how far architectural compression and prompt reuse reduce bytes per token. The most likely outcome is not “KV cache replaces HBM,” but a thicker hot-to-cold memory hierarchy: **HBM for active decode, DRAM for warm reuse, SSD for persistence and spill, remote tiers for cluster-wide sharing**. citeturn25search0turn22search8turn25search1turn15search13turn24search2turn24search3

[speculation] Over the next two to three years, the most investable thesis is that agentic and multimodal workloads make **context state** a first-class infrastructure problem. If that happens, value should accrue disproportionately to accelerator vendors with more memory bandwidth, memory suppliers with HBM exposure, and serving platforms that can keep cache hit rates high across replicas and clusters. Over five-plus years, the bigger uncertainty is architectural: if hybrids or non-attention sequence models materially reduce or sidestep KV state, parts of the current cache stack could be compressed into a lower-margin middleware layer. Today, though, the evidence still says the transformer family remains dominant, and the immediate commercial race is about managing KV better, not escaping it. citeturn16search0turn16search1turn5view0turn4view2turn15search1

## Part A Foundations

**What the KV cache is.** [established] In decoder-only transformers, each new token attends to all earlier tokens. Attention works by projecting token representations into **queries** \(Q\), **keys** \(K\), and **values** \(V\). For a current token, the query is compared with all stored keys to get attention weights, and those weights are used to mix the stored values into the next representation. During autoregressive generation, previously computed keys and values do not change, so engines cache them layer by layer. That is the KV cache. The transformer paper established the Q/K/V mechanism; systems papers such as vLLM operationalized the cache as the critical serving state. citeturn16search3turn13search0

**Why it is necessary.** [established] Without KV caching, the model would recompute K and V for the entire prefix every time it generated one more token. For a prompt of length *L* generating *T* more tokens, that means repeated work over prefixes of length *L, L+1, …, L+T-1* rather than a one-time prompt pass plus incremental updates. In practice, this would destroy throughput and latency. That is why all major serving stacks expose or assume KV reuse, and why public APIs now expose prompt/context caching as a product feature rather than a hidden implementation detail. citeturn13search0turn17search9turn10search2turn30view2turn31view2

**Prefill versus decode.** [established] Prefill processes the input prompt in parallel, creates the first full KV cache, and determines time-to-first-token. Decode then appends tokens one step at a time, reading model weights and the accumulated KV history for each new token. DistServe and BentoML’s handbook describe prefill as compute-heavy and decode as memory-bandwidth-heavy, which matches hardware intuition: prefill uses wide matrix multiplies with good arithmetic intensity, while decode often spends more time waiting on memory movement than on math. Flash-Decoding’s entire premise is that long-context decode is bottlenecked by loading K/V fast enough. citeturn37search9turn37search3turn37search22turn37search0

**Why “input is often the expensive part” even when APIs price input cheaply.** [evidence-backed inference] This statement is true in one sense and misleading in another. It is true for **uncached long prompts** because prefill must process every input token through all layers before the first answer token can appear, so long system prompts, large retrieved contexts, and full conversation histories can dominate TTFT. It is misleading as a universal rule because decode can dominate total serving cost for long outputs or decode-heavy workloads. Commercial price sheets also reflect product strategy, not pure marginal hardware cost: providers explicitly discount cached/reused prefixes, and all three major API vendors now sell some form of prompt or context caching because repeated input is operationally expensive enough to optimize. OpenAI says prompt caching can cut latency by up to 80% and input cost by up to 90%; Anthropic prices cache reads at 0.1× base input price; Google prices context caching separately and even charges a storage fee in some Gemini tiers. citeturn37search9turn37search0turn32view2turn30view1turn31view1

**Concrete size math.** [established] For a conventional cached-attention model, raw KV bytes are approximately:

```text
KV cache size ≈ layers × KV_heads × head_dim × 2 × batch × sequence_length × bytes_per_element
```

For models using GQA or MQA, use **KV heads**, not total query heads. For MLA-style models, the formula changes because what is cached is the compressed latent state, not full per-head K/V tensors. citeturn4view2turn1search1turn5view0turn9view5

**Worked example: Llama 3.1 70B.** [established] Meta’s published config for Llama 3.1 70B lists **80 layers, 64 attention heads, 8 KV heads, head dimension 128, and 131,072 max positions**. Plugging those into the BF16 formula yields about **320 KiB per token** of raw KV state. That is about **2.5 GiB at 8K**, **10 GiB at 32K**, and **40 GiB at 128K** for one sequence. Since the cache also scales with concurrency, **32 simultaneous 32K sessions** are already about **320 GiB** of raw KV data, and **10 simultaneous 128K sessions** are about **400 GiB**. Those figures are large enough to outrun a single GPU’s HBM even before weights, workspace, and fragmentation are counted. citeturn4view2turn19search0turn19search1

**Worked example: DeepSeek MLA.** [established] DeepSeek-V2 introduced **Multi-head Latent Attention** and officially reported that it **reduces the KV cache by 93.3%** relative to DeepSeek 67B while raising max generation throughput by **5.76×**. DeepSeek-V3 continues to use MLA. Its public config lists **61 layers, KV latent rank 512, and an extra RoPE key dimension of 64**; in the reference implementation, the RoPE key is shape `[bsz, q_len, 1, 64]`, which indicates that the cached MLA state is compact relative to full per-head tensors. A rough BF16 latent-cache estimate from those dimensions is on the order of **8.6 GiB for 128K tokens**—far below the Llama 3.1 70B BF16 GQA example above—but that should be treated as an **architectural estimate**, not a generic engine guarantee, because not every runtime stores or kernels MLA identically. DeepSeek’s FlashMLA repository exists precisely because specialized kernels are needed to realize the architecture’s intended efficiency in production. citeturn5view0turn8view0turn35view1turn35view3turn36search0

**Why the memory wall bites.** [established] The KV problem is really two problems: **capacity** and **bandwidth**. Capacity fails first when long contexts and high concurrency exhaust available HBM. Bandwidth fails continuously during decode because every token step has to stream model weights and the relevant KV blocks fast enough to keep the math units busy. H200’s 141 GB / 4.8 TB/s and MI300X’s 192 GB / 5.325 TB/s show how much the accelerator market has moved toward memory-rich inference parts, not just denser tensor cores. citeturn19search0turn19search1turn37search0

**Lost in the middle and attention patterns.** [established] Long-context behavior is not uniform across positions. The “Lost in the Middle” paper found many models perform best when relevant information appears near the beginning or end of long contexts and worse when it is buried in the middle. StreamingLLM identified **attention sinks**, showing that initial tokens often retain disproportionate attention mass even when semantically unimportant. H2O, SnapKV, PyramidKV, and related eviction schemes build on exactly these observations: some tokens matter much more than others, and “importance” varies across layers and positions. This is the conceptual bridge between long-context evaluation and KV-cache pruning. citeturn10search0turn10search1turn11search3turn11search2turn12search1

## Part B The research and technique frontier

**Architectural KV reduction.** [established] **MQA** reduces all heads to a single shared KV head; its canonical paper is Noam Shazeer’s “Fast Transformer Decoding: One Write-Head Is All You Need.” **GQA** generalizes that idea by using a small number of KV groups rather than only one, recovering much of multi-head quality while sharply shrinking cache. Google/DeepMind’s GQA paper showed multi-head checkpoints can be converted efficiently, and GQA is now mainstream in production open models such as Llama 3.x, where 64 query heads share only 8 KV heads. **MLA**, introduced by DeepSeek, goes further by caching a compressed latent representation instead of full per-head K/V. These three techniques are the most mature because they reduce KV at the model-design level and therefore avoid the runtime fragility of downstream pruning. MQA and GQA are clearly **deployed**; MLA is **deployed but specialized**, with DeepSeek’s FlashMLA kernels illustrating that software support lags behind the architectural idea. citeturn1search1turn1search2turn4view2turn5view0turn36search0

**Compression and quantization.** [established] **KIVI** showed that asymmetric **2-bit** KV quantization can preserve quality surprisingly well by quantizing keys per channel and values per token; **KVQuant** pushed further with 3-bit-style methods, pre-RoPE key quantization, and outlier-aware schemes, reporting near-lossless quality in many settings and context lengths up to 1 million on Llama-7B-class setups. [research frontier] Google Research’s **TurboQuant** is a newer training-free compression line that claims at least **6×** KV memory reduction and up to **8×** attention-logit speedup on H100-class evaluation for some workloads, but its practical landing zone in mainstream open-source serving stacks is still emerging. [deployed] Meanwhile, **FP8 KV cache** has crossed from paper to production: vLLM documents it as a first-line optimization, and NVIDIA TensorRT-LLM supports FP8 and NVFP4 KV cache paths. The trade-off is straightforward: lower bits improve capacity and throughput, but very low-bit schemes still face quality-risk tails, calibration complexity, and limited engine integration relative to FP8. citeturn11search1turn11search0turn27search4turn29search0turn29search3turn26search8

**Eviction, streaming, and sparsity.** [established] **StreamingLLM** keeps a small sliding window plus a few sink tokens, enabling effectively unbounded streaming without full-cache retention. **H2O** keeps “heavy hitter” tokens plus recency. **SnapKV** infers likely future-important positions from an observation window. **Scissorhands** argues for a “persistence of importance” hypothesis and reports 2–5× KV compression without fine-tuning. **PyramidKV** and **PyramidInfer** exploit the observation that higher layers can often get away with smaller retained context than lower layers. [research versus deployed] This entire family is vibrant academically, but deployment is patchier than the paper count suggests because pruning interacts with model quality, safety, determinism, and benchmark realism. The common trade-off is simple: the more aggressively you throw away tokens or channels, the more workload-specific your win becomes. citeturn10search1turn11search3turn11search2turn12search3turn12search1turn12search2

**Reuse and sharing.** [established] **PagedAttention** is the major systems breakthrough that made large-scale KV management practical in open-source serving: it stores KV blocks in non-contiguous paged memory, reducing fragmentation and enabling efficient sharing. **RadixAttention** in SGLang adds an explicit reuse structure for common prefixes across requests. **Automatic prefix caching** is now standard in vLLM, SGLang, TensorRT-LLM, TGI, and multiple hosted APIs. [frontier] **Non-prefix reuse** is where the frontier is moving: **CacheBlend** showed that precomputed KV chunks can be reused even when they are not an exact prefix, especially for RAG. But this is still much less mature operationally than prefix caching because it must recover inter-chunk attention correctly, manage retrieval granularity, and prevent cache-fusion bugs and tenant-isolation issues. citeturn13search0turn13search1turn13search2turn17search9turn29search9turn28search0turn28search3

**Offload and disaggregation.** [established] When HBM is not enough, systems increasingly offload KV blocks to **CPU RAM**, **local disk/NVMe**, or **remote/shared storage**. LMCache documents CPU RAM and local storage tiers, while NVIDIA Dynamo exposes KV offload through vLLM connectors. Mooncake is the clearest large-scale proof point for **KV-cache-centric disaggregation**: it separates prefill and decode clusters and builds a distributed KV pool from underused CPU, DRAM, and SSD resources, reporting large capacity gains under real Kimi workloads. GPUDirect Storage provides the underlying direct path that makes GPU–storage movement less CPU-bound. [disagreement] Here credible sources diverge: systems papers and product docs show offload works, but they also make clear it works best when cache reuse is high and cold misses are infrequent. Offload is therefore best understood as a **hierarchical memory strategy**, not a magic replacement for HBM. citeturn14search12turn14search0turn15search13turn14search9turn26search15turn15search4

**Architecture-shift risk.** [established] KV cache is a transformer problem, so any serious move away from transformer-style attention would change the value stack. **Mamba** and related state-space models replace growing token-by-token KV state with a compact recurrent state. **Jamba** and other hybrids interleave transformer and Mamba blocks rather than abandoning attention completely. [evidence-backed inference] The current industry consensus is not that attention is dead; it is that pure transformers are being selectively mixed with more state-efficient mechanisms where they help. In production frontier models, attention-reduced but still transformer-like designs—GQA, sparse attention, MLA, hybrid stacks—are far more mature today than a wholesale move to non-attention LLMs. That means KV remains central for the next few years even if its exact form changes. citeturn16search0turn16search1turn5view0turn36search0

**Trajectory.** [evidence-backed inference] The strongest near-term methods are the ones closest to deployment reality: **GQA/MQA/MLA**, **FP8 KV cache**, **PagedAttention/RadixAttention**, **prompt caching**, and **prefill/decode disaggregation**. The most important open problems are harder ones: non-prefix reuse at cluster scale, quality-safe eviction under heterogeneous workloads, multimodal cache representation, cache-aware routing across replicas, and security/isolation of shared cache infrastructure. citeturn4view2turn29search0turn13search0turn13search2turn15search1

## Part D Industry impact

**Memory.** [evidence-backed inference] Rising KV pressure is clearly bullish for **HBM capacity and bandwidth**, because active decode must stay hot. The harder question is whether it is also meaningfully bullish for **DRAM** and **NAND/SSD**. The bull case is strong: long-context agents, repeated context, and distributed serving all create reasons to keep warm KV outside HBM but close enough to reuse, and vendors from Samsung to Kioxia to WD are now openly designing for this use case. The bear case is also strong: better architectures \(GQA, MLA\), better quantization \(FP8 and lower\), and better prefix reuse all reduce the number of bytes needed per useful token. Netting those together, the likeliest outcome is **bandwidth demand remains structurally hot in HBM, while capacity demand fans out into DRAM and SSD as a hierarchical “context memory” layer rather than a one-for-one substitution for HBM**. citeturn25search0turn22search8turn25search1turn29search0turn5view0

**Semiconductors.** [evidence-backed inference] KV pressure is shifting semiconductor design toward **memory-centric inference architectures** and toward a clearer hardware split between prefill and decode. NVIDIA’s Dynamo explicitly treats distributed KV management as a first-class software layer; Marvell and Astera are selling CXL for pooled memory; specialist silicon vendors emphasize on-chip SRAM or dataflow. Architecturally, the watchword is no longer just FLOPS; it is **bytes moved per useful token**. That favors accelerator and interconnect vendors that can either widen the hot tier or shrink the amount of state that must stay in it. citeturn15search1turn24search2turn24search3turn20search8turn20search11

**Model training.** [evidence-backed inference] KV cache is technically an inference artifact, but it feeds back into training choices. Long-context training makes serving economics more painful unless the architecture is cache-efficient. DeepSeek’s MLA is exactly this kind of co-design choice: an attention mechanism chosen partly for inference efficiency. Reinforcement-learning and post-training loops also matter because chain-of-thought, tool traces, and long conversational histories make people actually use the long context they train for. The result is a growing bias toward architectures whose training-time decisions are judged partly by future inference memory pressure. citeturn5view0turn6search2

**Inference economics.** [established] Better KV management lowers **TTFT**, improves **throughput**, and often improves **margins** because it raises the number of concurrent useful sessions a cluster can support. Prompt caching is the clearest commercial proof: OpenAI, Anthropic, Google, and Fireworks all turned it into a concrete pricing/latency feature. [evidence-backed inference] For providers, the unit-economics implication is subtle: good caching can reduce the compute cost of repeated input dramatically, but it can also stimulate more usage by making agentic and long-context patterns economically viable. That is a classic efficiency-rebound setup. citeturn32view2turn30view1turn31view1turn17search1

**Agentic workflows.** [established] Agents are the structural driver of KV pressure. They accumulate long histories, repeat large system prompts and tool schemas, revisit shared background knowledge, and spawn side-calls that often share significant prompt structure. Anthropic’s docs explicitly frame one-hour prompt caching as useful when agent side-tasks may stretch beyond five minutes; OpenAI’s prompt-cache docs explicitly cover tools and structured outputs; Google’s context caching exists because repeated input is common enough to justify a storage-priced feature. [evidence-backed inference] If agents become mainstream, the important metric will not be tokens in isolation but **reused-context ratio**: what fraction of a workflow’s input can be served from cached state rather than recomputed prefill. That favors systems with prefix-aware routing, shared caches across replicas, and chunk/non-prefix reuse for RAG-style knowledge access. citeturn30view3turn32view0turn32view3turn31view2turn15search2turn28search0

**Multimodal.** [established] Images and video worsen the KV problem because they typically expand into many tokens or token-like units once injected into a decoder’s context, and vendors now treat multimodal repeated input as cacheable. OpenAI says prompt caching can cover images; Google prices text/image/video context caching explicitly; SGLang’s paper says RadixAttention is compatible with multimodal models and hashes images for reuse keys. [evidence-backed inference] Multimodal therefore raises both the pain and the payoff: the cache gets much bigger, but repeated visual context is also especially worth caching because the prefill is so expensive. citeturn32view0turn31view1turn13search5

## Part E Outlook and debates

**Does KV demand explode or get optimized away?** [evidence-backed inference] The bull case for “explode” is straightforward: cheaper inference induces more usage, and the usage that grows fastest is exactly the usage that is context-heavy—agents, copilots, multimodal assistants, and long-horizon enterprise workflows. Prompt caching makes repeated context cheaper, which can increase the volume of repeated-context applications. The bear case is equally powerful: KV bytes per token are already falling through GQA, MLA, FP8 KV, better routing, and possible future hybrid architectures. The evidence that would settle the debate is not another synthetic benchmark; it is production telemetry on **bytes of active/warm/cold context per generated token** across real deployments over time. citeturn32view2turn30view1turn31view1turn29search3turn5view0

**Is KV cache the next Big Data?** [evidence-backed inference] The “for” argument is that KV is becoming a reusable, routeable, monitorable substrate for inference. LMCache explicitly frames KV as reusable AI-native knowledge; llm-d and Mooncake both move toward cross-node cache coordination; cloud vendors are building cache-aware fabrics and routing policies. The “against” argument is deeper and, in my view, stronger: **KV cache is model-specific, precision-specific, tokenizer-specific, prompt-order-specific, and often short-lived**. It is not a portable enterprise data asset in the way logs, tables, or embeddings can be. It is more like highly optimized intermediate state than like a durable database. So the analogy is useful for systems design, but overstated as an asset thesis. citeturn14search12turn15search10turn26search15turn34search7

**A CDN for KV cache.** [speculation] The idea is real in a narrow sense and overhyped in a broad sense. It is real when there are repeated prefixes or reusable chunks across sessions and nodes: that is exactly what llm-d, LMCache, Mooncake, and cache-aware routing are trying to do. It is overhyped if imagined as a universal semantic cache that can serve arbitrary near-matching prompts anywhere on the network with negligible penalties. Network distance, tenant isolation, exact-match requirements, and recomputation needs still matter. The most plausible near-term form is **enterprise-local distributed KV fabrics**, not a global internet-scale CDN. citeturn15search2turn15search10turn26search15

**Where durable value accrues.** [evidence-backed inference] The highest-confidence answer is **hardware plus scale-out systems software**. Hardware wins because HBM and bandwidth are still scarce and expensive, and because every efficiency gain upstream tends to make latency-sensitive inference workloads more viable, not less. Systems software wins because open-source engines have already shown that local single-node optimization is not enough; prefix-aware routing, offload policies, disaggregated prefill/decode, and cluster-wide reuse are where remaining slack lives. Model labs still capture value when they redesign the problem itself—DeepSeek with MLA is the clearest case—but a lot of that value may leak into the serving ecosystem as techniques diffuse. citeturn19search0turn19search1turn15search1turn15search13turn5view0

**Scenario sketch.** [speculation] Over the next **two to three years**, expect FP8 KV cache, prefix caching, cache-aware routing, and disaggregated prefill/decode to become default in serious serving stacks; HBM-rich inference SKUs and CXL/SSD context tiers should remain supply-and-design focal points; and more model labs will make explicit architecture-for-inference choices rather than treating serving as downstream optimization. Over **five-plus years**, two futures compete: either the transformer family remains dominant and KV evolves into a mature hierarchical context-memory subsystem, or hybrid/state-space designs reduce cache centrality enough that some of today’s middleware value migrates back into model architecture. The base case today is still the former. citeturn29search3turn15search1turn24search2turn16search0turn16search1

**Open questions and limitations.** [established] Several important points remain genuinely unsettled. Independent production benchmarks for many specialist hardware vendors and low-bit KV schemes are still thinner than vendor messaging. “How much NAND wins” is especially uncertain because software reuse and architectural compression can hollow out the addressable bytes even as agentic usage raises total workload volume. And there is no single canonical metric for KV efficiency that spans all engines, models, quantizations, and multimodal formats. Those are not gaps in this report so much as live open problems in the field itself. citeturn20search10turn27search4turn25search0turn22search8

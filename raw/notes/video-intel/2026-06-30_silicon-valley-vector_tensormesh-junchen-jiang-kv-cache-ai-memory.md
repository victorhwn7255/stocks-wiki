## TL;DR

1. **The real inference cost is in the INPUT (prefill), not the output.** People think input tokens are cheap because pricing makes them cheap and because users spend ~90% of their *staring* time watching output stream out (output is sequential, can't parallelize). But in *true compute* (FLOPs/token), processing an input token ≈ generating an output token — and in real agentic/RAG workloads, input is often **10×+ longer** than output. So aggregate prefill cost is as big or bigger; provider "cheap-input" pricing reflects pricing balance, not cost.
2. **KV Cache = the model's reusable memory.** A model is stateless and throws away its internal understanding (the KV-cache tensors) after each call. Tensormesh/LMCache preserves and reuses it across calls → avoids recomputing prefill on repeated/shared context.
3. **"Trade storage for compute" (以存代算).** If storage is fast enough (CPU RAM, local SSD, GPUDirect Storage), fetching cached KV is *both cheaper and faster* than recomputing — a rare win-win. Tier hot KV → HBM, colder → SSD/remote.
4. **Counterweight to the memory-bull thesis (his book):** storage being expensive is a supply-demand imbalance that "will come down" once capacity catches up — though **"not fast, because storage vendors' orders are already booked ~2 years out."** The smart economic move is software efficiency, not buying more memory.
5. Color: **Chinese open-source models (DeepSeek '25, Zhipu GLM '26) are the best open-source models — US big-cos still use them**; hybrid Mamba/Transformer is the architecture consensus (Qwen 3.5 mostly Mamba, ~¼ layers still attention); M&A consolidation ("a big one every month, soon every week").

## Key claims + numbers (all soft; grouped by topic)

**Prefill cost — the load-bearing, counter-intuitive claim:**
- > *"If you look at the real compute cost — not the pricing — processing an input token and generating an output token cost about the same. … Input isn't cheaper; the compute is roughly equal."* (~03:30) `[opinion]` — true per-token FLOPs ≈ equal; input *feels* cheap only because output is sequential and users wait on it.
- > *"In real use cases the input can be 10-plus times longer than the output — tens of thousands of input tokens, a few thousand output tokens — yet users spend only ~10% of their time on input processing and ~90% watching output."* (~05:00) `[opinion]` — so aggregate prefill cost ≥ output cost; cheap-input pricing = a provider balancing choice, not cost.
- Because models are **stateless**, agent/chatbot context must be re-fed every turn → input grows without bound, output doesn't. (~06:30) `[opinion]`

**KV Cache — three layers (the framework centerpiece):**
- **L1 (black box):** storable compute state, reuse as-is. "Many industry companies do this." `[opinion]`
- **L2 (academia ahead):** KV cache carries **semantic/attention information** → you can modify the values (lossy compression, updates) as long as you don't change the attention → still reusable, even across *different text with similar meaning* and *across different models*. "Treat it as a white box." `[forward]`
- **L3 (most interesting; industry rarely gets it):** deliberately *change* the semantics — "tell the model pay more attention to this, less to that" → can *raise* output accuracy/quality. Output = f(weights, KV-cache). `[forward]`
- > *"KV Cache is the Big Data of the AI era."* (~10:00, recurring) `[opinion]` — the moat is that L1 needs infra/GPU engineers, L2-L3 need ML researchers, and almost no one spans both; big-cos have all three skill types but "can't put them in one team / lack the vision and focus" (cites Sam Altman on why startups beat incumbents — attention is too dispersed). 
- > *"The KV-cache moment may come earlier than I thought — possibly before the end of this year [2026]."* (~13:00) `[forward]`

**Storage economics & the cycle ("trade storage for compute"):**
- If storage is fast enough (CPU RAM / local SSD / GPUDirect Storage), fetching cached KV **saves GPU compute AND is faster than recomputing** — "both cheaper and faster." Colder/remote storage → cut cost but add latency. KV-cache size shrinks via compression, so "as long as the network isn't too slow, caching is generally beneficial." (~30:00) `[opinion]`
- > *"Storage is expensive because supply and demand are imbalanced. Once capacity comes up the storage sells and prices come down — not fast, because many storage vendors' orders are already booked two years out — but long-term it comes down."* (~26:00) `[forward/opinion]` ← **the book-talking line; note it both (a) undercuts memory-durability and (b) independently corroborates the ~2-year booked-out backlog.**

SSD prices may be approaching RAM now — scarce, so priced up — so SSD may not be economical; but for very large models/KV-cache you must use SSD, and it still helps a lot." Their business is **decoupled from the storage tech** — "whatever storage you have, we store KV cache most efficiently." (~31:00) `[opinion]`
